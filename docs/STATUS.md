# STATUS.edn

`STATUS.edn` is the source-of-truth capability manifest for the README
"## Current status" section. The section is **generated**, not hand-written:
it is regenerated by `bb status` and verified in CI by `bb status --check`.

## Why

The README status section used to drift from reality — capabilities were
claimed as "implemented" that were only partially merged, and the
"not yet implemented" list went stale as features landed. `STATUS.edn`
makes the section truthful by construction: the manifest is the only input,
and CI fails if README disagrees with it.

## The `status` task

```bash
bb status           # regenerate the README section from STATUS.edn
bb status --check   # CI gate: exit 1 if README is out of sync or EDN is invalid
```

`bb status` parses `STATUS.edn` with `clojure.edn/read` (no extra deps),
validates the schema below, and rewrites the bytes between two sentinel
markers in `README.md`:

```markdown
<!-- BEGIN STATUS -- generated from STATUS.edn by `bb status`. Do not edit by hand. -->
…generated…
<!-- END STATUS -->
```

The `## Current status` heading stays **outside** the markers so the section
anchor is stable. Do **not** edit the content between the markers by hand —
`bb status --check` will fail and tell you to regenerate.

## Schema

`STATUS.edn` is an EDN map with a single top-level key `:tesseraft`:

```edn
{:tesseraft
 {:generated-from "STATUS.edn"
  :readme-section "## Current status"
  :capabilities
  {<keyword> {:status < :implemented | :partial | :not-implemented >
              :summary "<one-line human summary>"
              :evidence [<repo-relative path or symbol> …]
              :gap "<optional, only for :partial>"}}
  :not-yet-implemented [<string> …]}}
```

Validation enforced by `scripts/generate_status.clj`:

- top-level `:tesseraft` key present and a map;
- `:capabilities` is a map;
- every capability has `:status` in `#{:implemented :partial :not-implemented}`;
- every capability has a non-empty `:evidence` vector.

Existence checks on evidence paths are intentionally **not** enforced in v1
(to avoid false failures on renamed files); keep evidence honest anyway.

## Rendering

Capabilities are grouped by status — `:implemented`, then `:partial`, then
`:not-implemented` — each rendered as a bulleted entry with its summary,
optional `:gap` note (for partials), and evidence refs. The
`:not-yet-implemented` list is rendered as a final plain bullet list.

## When to update `STATUS.edn`

Whenever a change adds, removes, or alters a Tesseraft capability, update
`STATUS.edn`, run `bb status`, and commit `STATUS.edn` + `README.md` in the
**same commit** as the source change. `bb status --check` must pass locally
before opening the PR. The agent PR-authoring prompt templates encode this
obligation (see `examples/*/prompts/pr-draft.md.tmpl`).

## CI

`.github/workflows/ci.yml` runs `bb status --check` after `bb test`, so
status drift fails CI. `scripts/test.sh` also runs the check so `bb test`
catches drift before push.

## Truthfulness policy

`STATUS.edn` records the **actual** state on `main`, not aspirations. A
capability that is only partially merged (e.g. a schema enum without runtime
support, or an example without the runner mode) is `:partial` with a `:gap`
note — never rounded up to `:implemented`. See the `:gap` fields in
`STATUS.edn` for current examples (blocked run state, mock executor).
